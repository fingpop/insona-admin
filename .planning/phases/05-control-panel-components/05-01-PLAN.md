<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client UI -> /api/devices/control | Control commands from browser — device ID, action, value, meshId are user-controlled |
| Slider value -> gateway | Raw numeric values (0-100) sent directly to gateway without server-side validation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Tampering | onControl calls | mitigate | Server-side `/api/devices/control` validates device ownership, meshId format, and value ranges (0-100) before forwarding to gateway |
| T-05-02 | Tampering | Slider direct-send | accept | Values clamped 0-100 in UI, server validates ranges — no injection vector via range input |
| T-05-03 | Repudiation | Control commands without confirmation | accept | Internal network, no audit requirement; control commands are idempotent (same value = same state) |
| T-05-04 | Denial of Service | Rapid slider events (mouseUp spam) | mitigate | onControl is async + controlling state disables buttons — natural rate limiting prevents gateway flood |

Note: This phase modifies existing control UI that already calls the same `/api/devices/control` endpoint. No new API routes or authentication changes. Server-side validation from Phase 3/4 remains in place.
</threat_model>
